# 인프라 로드맵

가맹점 50개 베타 테스트 + 향후 비가맹점 확장을 목표로 정리한 장기 계획.
2026-07-09 논의 기준. 완료된 항목은 체크, 나머지는 우선순위 순.

## Phase 1 — 가맹점 50개 베타 시작 전 (긴급도 높음)

- [x] SQLite WAL 모드 + busy_timeout 적용 (동시 쓰기 락 경합 방지)
- [x] Playwright 헤드리스 브라우저 동시 실행 개수 전역 제한 (`browser_limit.py`, 기본 4개)
- [ ] **백업 전략 마련** — SQLite 파일(`inventory.db`)이 Render 디스크 하나에만 있음.
      정기 백업(예: 매일 새벽 크론으로 S3/외부 스토리지에 사본 저장) 필요.
      디스크 장애 시 복구 방법이 지금은 없음.
- [ ] **배포 타이밍 원칙 정하기** — 배포마다 짧은 502가 발생함(Render 무중단 배포
      아님). 영업시간(아침 발주 몰리는 시간) 피해서 배포하는 걸 팀 규칙으로.

## Phase 2 — 베타 진행 중 안정화

- [ ] **에러 모니터링 도입** (Sentry 등) — 지금은 문제가 생기면 사용자가 말해줘야
      알 수 있는 구조. 가맹점 50개 규모에서는 조용히 묻히는 실패가 늘어남.
- [ ] **커스텀 도메인으로 전환** — `.onrender.com`은 브랜드 페이지/가맹점 접점에는
      신뢰도상 좋지 않음. 도메인 구매 후 Render에 연결(SSL 자동).

## Phase 3 — 비가맹점(외부 고객) 확장 전 필수

- [ ] **DB를 PostgreSQL 등으로 마이그레이션** — SQLite는 예측 불가능하게 늘어나는
      공개 서비스 동시 접속에 구조적 한계.
- [ ] **Playwright 작업을 별도 워커/큐로 분리** — 지금은 요청-응답 안에서 브라우저를
      직접 띄우는 구조. 외부 트래픽까지 감당하려면 백그라운드 워커 풀로 분리 필요.
- [ ] **셀프서브 온보딩 플로우** — 지금은 대표님이 텔레그램에서 한 명씩 승인하는
      구조. 비가맹점 대상이면 이 병목이 못 버팀.
- [ ] **결제/구독 인프라** — 비가맹점에 유료 서비스로 낼 경우 처음부터 필요.
- [ ] **이용약관 / 개인정보처리방침** — 도매처 스크래핑, 쿠팡 파트너스 연동,
      사업자번호 등 개인정보 수집이 있어 외부 사용자 대상이 되는 순간 법적 문서 필요.
